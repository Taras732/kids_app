## Архітектура

Школярик використовує **2 hosted Supabase проекти** (без локального Docker stack):

- `shkolyaryk-dev` — для щоденної розробки та QA
- `shkolyaryk-prod` — для production EAS builds

Обидва проекти — регіон **Frankfurt (eu-central-1)**.

Експо-app бере конфіг з `.env` (dev) або EAS secrets (prod).
CLI-операції (міграції, config) виконуються проти одного з проектів залежно від `supabase link`.

## One-time setup для нового розробника

### 1. Встановити Supabase CLI

```bash
# Варіант A — Scoop (Windows, preferred)
scoop install supabase

# Варіант B — npm global
npm install -g supabase

# Verify
supabase --version  # має бути ≥1.200.0
```

### 2. Отримати credentials

Попроси lead-а (Taras) додати тебе до обох проектів у Dashboard.
Потім:

```bash
supabase login  # відкриє браузер для OAuth
```

### 3. Link локального репо до dev-проекту

```bash
cd d:/Dev/shkolyaryk
supabase link --project-ref <DEV_PROJECT_REF>
```

Project refs (запитай lead-а, не коміть у репо):
- dev: `<dev-ref>`
- prod: `<prod-ref>`

### 4. Створити `.env`

```bash
cp .env.example .env
# Відкрий .env і заповни EXPO_PUBLIC_SUPABASE_URL + ANON_KEY з Dashboard → Project → Settings → API (dev project!)
```

### 5. Перевірити міграції синхронні з БД

```bash
supabase migration list
# Має показати 0001, 0002 як applied — без pending
```

Якщо пусто або розсинхрон — зверни до lead-а, **не роби `db reset`** на hosted проекті.

## Daily workflow

### Додати нову міграцію

```bash
# 1. Створити файл
supabase migration new <short_description>
# Редагуй supabase/migrations/YYYYMMDDHHMMSS_<short_description>.sql

# 2. Задеплоїти на dev
supabase db push

# 3. Перевірити в Dashboard → Database → Migrations

# 4. Закомітити
git add supabase/migrations
git commit -m "feat(us-XXX): описати що додано"
```

### Переключитись на prod (тільки для релізу)

```bash
supabase link --project-ref <PROD_PROJECT_REF>
supabase migration list     # перевірити поточний стан prod
supabase db push            # deploy
# Після цього ОБОВ'ЯЗКОВО переключись назад на dev:
supabase link --project-ref <DEV_PROJECT_REF>
```

### Оновити email templates / auth config

1. Редагуй `supabase/config.toml` або `supabase/templates/*.html`
2. `supabase config push`
3. Verify у Dashboard → Auth → Email Templates

## Env switching для Expo app

```bash
# Dev (default)
npx expo start -c                     # читає .env

# Prod build (EAS)
eas build --profile production        # читає EAS secrets (не локальний .env.production)
```

Для локальної перевірки prod-значень без EAS build:
```bash
cp .env.production .env
npx expo start -c
# ⚠️ Після тесту поверни .env назад на dev!
```

## Troubleshooting

### `supabase link` fails з "access denied"

- Перевір `supabase login` — має бути authorized
- Lead має додати тебе як organization member у Dashboard → Organization Settings

### `migration list` показує pending що не існує в `supabase/migrations/`

Це означає БД попереду коду. Варіанти:
- `supabase db pull` — згенерувати міграцію з поточного стану БД
- Або попроси lead-а дати SQL-файл що він деплоїв manually

### Email confirmation link не відкриває app

Перевір у Dashboard → Auth → URL Configuration:
- **Site URL** = `shkolyaryk://`
- **Redirect URLs** містить `shkolyaryk://auth/callback`

Для web-тесту можна тимчасово додати `http://localhost:8081`.

### `npx expo start` показує `placeholder.supabase.co`

`.env` не зчитався. Перевір:
- Файл називається саме `.env` (не `.env.local`)
- Запущено з `-c` flag (очистка Metro cache)
- Змінні починаються з `EXPO_PUBLIC_` (інакше не потрапляють у client bundle)

## Rollback міграції

Supabase не підтримує автоматичний rollback. Якщо треба відкотити:

1. Створи нову міграцію що робить зворотні зміни (`DROP TABLE`, `ALTER ... DROP COLUMN`)
2. `supabase db push`
3. Для позначення старої міграції як reverted (якщо треба переписати):
   ```bash
   supabase migration repair --status reverted <version>
   ```

**Правило:** на prod ніколи не робимо `migration repair reverted` без backup. Завжди — нова forward-міграція.
